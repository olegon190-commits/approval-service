# DESIGN — approval-service

## Модель данных

Три таблицы:

**approval_requests** — заявки на согласование. Хранит внешние идентификаторы (`source_id`, `reviewer_user_ids`, `created_by`) как строки — сервис не знает о соседних сущностях и не хранит их данные. Статусная модель: `pending → approved | rejected | cancelled`. Финальные статусы неизменны.

**audit_logs** — след каждого успешного изменения: кто (`actor_user_id`), что (`action`), когда и с какими деталями. Пишется в той же транзакции что и изменение.

**outbox_events** — события для будущих интеграций (outbox pattern). Каждое изменение оставляет событие (`approval_request.created`, `.approved`, `.rejected`, `.cancelled`) в той же транзакции. Отдельный воркер (вне рамок задания) сможет публиковать их в брокер и помечать `published_at`.

## Границы сервиса

Сервис отвечает только за процесс согласования. Публикации, сценарии, пользователи и workspace — внешние сущности, передаются как идентификаторы без валидации их существования. Это осознанное решение: проверка принадлежности `reviewer_user_ids` к workspace — ответственность вызывающего сервиса или API-гейтвея.

## Изоляция workspace

Двухуровневая проверка:
1. `X-Workspace-Id` из auth-контекста должен совпадать с `workspace_id` из URL — иначе 403.
2. Все запросы к БД фильтруются по `workspace_id` — заявка чужого workspace даёт 404, не раскрывая факт её существования.

## Обработка повторов (идемпотентность)

Клиент передаёт заголовок `Idempotency-Key` при создании. Если заявка с таким ключом уже есть в этом workspace — возвращается она же, дубль не создаётся. Ключ хранится в самой заявке, индекс по `(workspace_id, idempotency_key)`.

Компромисс: в текущей реализации возможна гонка при одновременных запросах с одним ключом (нет уникального constraint из-за nullable-поля). Для продакшена: partial unique index `WHERE idempotency_key IS NOT NULL` + обработка конфликта вставки.

Операции решения (approve/reject/cancel) идемпотентны по своей природе: повтор даёт 409, состояние не меняется.

## Финальные статусы

Проверка `_ensure_not_final` перед любым решением: заявка в статусе `approved`, `rejected` или `cancelled` не может перейти в другой финальный статус — возвращается 409 Conflict.

## События / интеграции

Outbox pattern: события пишутся в таблицу в той же транзакции что и изменение данных — гарантия что событие не потеряется и не появится без изменения. Следующий шаг (вне задания): воркер читает неопубликованные события и шлёт в Kafka/RabbitMQ/NATS.

## Безопасность данных в ответах

Публичные ответы собираются явно через `ApprovalRequestOut.from_model` — только допустимые поля. Ничего лишнего (внутренние ключи, идемпотентность, сырые данные) наружу не попадает. Логи и события содержат только идентификаторы и бизнес-поля — без секретов, токенов, email и URL.

## Известные компромиссы

1. **Гонка идемпотентности** — описана выше, решение известно.
2. **Нет пагинации списка** — для MVP достаточно, добавляется limit/offset тривиально.
3. **Миграции — plain SQL** — для боевого сервиса стоит взять Alembic; здесь SQL нагляднее и проще для ревью.
4. **`create_all` при старте** — удобно для локального запуска и тестов; в продакшене отключается, схема управляется только миграциями.
5. **Auth-заглушка доверяет заголовкам** — по условию задания; интерфейс (user, workspace, actions) совпадает с тем что даст реальный JWT middleware.
